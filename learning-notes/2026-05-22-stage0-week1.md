## 0. 왜 배우나 (동기)

TS에선 "내 코드가 어디서 도는가?"가 명확하다: `.ts →(tsc)→ .js →(node/브라우저)→ 실행`.
그런데 JVM 세계에 오면 **"JDK 21 설치", "JRE면 실행은 됨", "JVM 위에서 돈다", "바이트코드", "Temurin/OpenJDK/Oracle"** 가 한꺼번에 쏟아진다.
이 셋(JDK/JRE/JVM)을 모호하게 넘기면 나중에 *"왜 Docker엔 JRE만 넣지?", "왜 컨테이너 메모리랑 JVM 힙이 따로 놀지?"* 에서 계속 막힌다.
→ 1주차 목표: **"내 Kotlin/Java 코드가 어디서·무엇에 의해 실행되는가"를 TS 멘탈모델에 정확히 포개기.**

## 1. 실행 파이프라인 (TS와 대비)

```
[TS]   .ts   ──(tsc)──▶        .js   ──(node)──▶  실행
                ▲                        ▲
            컴파일러                  런타임/실행기

[JVM]  .java ──(javac)──▶  .class(바이트코드) ──(JVM)──▶  실행
.kt   ──(kotlinc)──▶
                ▲                                  ▲
         컴파일러(JDK 안)                       실행기
```

- `javac`/`kotlinc` = `tsc` 자리 (컴파일러)
- `JVM` = `node`(V8) 자리 (실행기)
- **흔한 오해 교정:** `.java → .class` 변환은 **JVM이 아니라 컴파일러(javac)**가 한다. JVM은 *그다음* `.class`를 실행한다.

### 바이트코드(.class)
- OS·CPU에 독립적인 중간 표현 → **같은 .class가 어디서든 JVM만 있으면 동일 동작** (Write Once, Run Anywhere).
- 사람이 읽는 텍스트가 아니라 바이너리. (`javap -c App.class`로 역어셈블해 구경은 가능 — Phase 2에서)

## 2. JDK / JRE / JVM (포함관계)

```
JDK (개발 키트)  ⊃  JRE (실행 환경)  ⊃  JVM (가상 머신/실행기)
```

| 약어 | 풀네임 | 역할 | TS/Node 비유 |
|---|---|---|---|
| **JVM** | Java **Virtual Machine** | 바이트코드를 실제 실행하는 엔진 | V8 엔진 |
| **JRE** | Java **Runtime Environment** | JVM + 표준 라이브러리 (실행 전용) | node 런타임 |
| **JDK** | Java **Development Kit** | JRE + 컴파일러(javac)·도구 (개발+실행) | node + tsc + 개발툴 |

- **컴파일하려면 → JDK** 필요 (javac 포함)
- **실행만 하려면 → JRE/JVM** 면 충분
- 실무: Docker 배포 이미지엔 JDK 대신 **JRE만** 넣어 용량 절감 (Phase 6에서 활용)

### 기억 트릭
- **V**irtual machine → 굴리는 엔진 (제일 안쪽)
- **R**untime → 실행(run)만 (중간)
- **D**evelopment → 개발(만들기)까지 (제일 바깥, 풀세트)

### Trade-off: JRE/JDK를 왜 나눴나
- **유리한 점:** 배포 환경(서버·Docker)엔 컴파일러가 필요 없으니 **JRE만** 넣으면 이미지가 절반 이하로 작아진다.
- **JDK가 필요한 때:** 개발 머신은 컴파일을 하므로 JDK 필수.
- **요즘 흐름:** Java 11부터 **별도 JRE 배포가 사라짐**. 대신 `jlink`로 필요한 모듈만 추려 경량 런타임을 만든다.
  → "JRE"는 *개념*으로 알아두되, 실무에선 JDK 안에서 다 해결하거나 jlink로 추리는 경우가 많다.

### JDK 배포판 (OpenJDK vs Temurin vs Oracle)
- JDK는 **"명세(스펙)"** 이고, 그 스펙을 구현한 **여러 배포판**이 있다. (npm 레지스트리는 하나지만 Node 배포는 nodejs.org/nvm/배포판마다 다른 것과 비슷)
- **OpenJDK**: 오픈소스 레퍼런스 구현. **Temurin**(=Eclipse Adoptium): 무료·실무 표준 배포판. **Oracle JDK**: 오라클 상용(라이선스 주의).
- 이 과정에선 **Temurin 21** 사용: `brew install --cask temurin@21`

## 3. `main` = 프로그램 진입점(entry point)

```java
public static void main(String[] args) { ... }
```

- JVM은 프로그램을 켤 때 **무조건 `main`부터 찾아 호출**한다 → `main`이 **프로그램의 시작점(진입점)**.
- JS/Node엔 이런 게 없다: `node app.js` 하면 파일이 위에서부터 그냥 실행됨. Java는 "**시작은 반드시 `main`에서**"라는 약속이 있다.
- `main`은 클래스 안의 `static` 메서드다(맞는 말). 단, 빈칸의 핵심 역할은 "프로그램의 **진입점**".

## 4. 컴파일 vs 파싱 (헷갈림 정리)

**파싱은 컴파일의 대등한 짝이 아니라, 컴파일 "안의 한 단계"다.**

```
소스 텍스트 → ① 렉싱(쪼개기) → ② 파싱(구조 만들기) → ③ 의미분석 → ④ 코드 생성
"val x = 1"   [val][x][=][1]      트리(AST)            타입체크 등    .class/.js/기계어
```

- **렉싱(lexing/토크나이징):** 텍스트를 토큰으로 **쪼갠다**. (← "쪼갠다"는 사실 이것)
- **파싱(parsing):** 쪼갠 토큰을 **문법 트리(AST)로 조립**해 "말이 되는 구조인지" 세운다.
  - 문장 비유: "나는 밥을 먹었다" → 쪼개기는 `[나는][밥을][먹었다]`, 파싱은 `주어+목적어+동사` 트리로 구조화.
  - 구조가 깨지면 여기서 **Syntax Error**.
- TS에서 매일 겪는 것: `tsc`/Babel/ESLint/Prettier 모두 내부에서 코드를 파싱해 **AST**를 만든다. 빨간 줄 `Syntax Error` = 파싱 실패.

### 정의 교정
| 처음 이해 | 교정 |
|---|---|
| 컴파일 = 고차원 언어 → 저차원(바이너리) | 거의 맞음. 단 결과가 꼭 바이너리는 아님(tsc→JS는 텍스트). "더 낮은/다른 표현으로 **번역**"이 정확 |
| 파싱 = 언어를 음절단위로 쪼갠다 | "쪼개기"는 **렉싱**. 파싱은 토큰을 **문법 트리로 조립·구조화**하는 단계 |

## 5. 미션 복기 (읽기 연습)

읽은 코드:
```java
public class Hello {
    public static void main(String[] args) {
        String name = "Arnold";
        System.out.println("Hello, " + name);
    }
}
```

| 미션 | 결론 |
|---|---|
| 실행 단계 | `.java →(javac)→ .class →(JVM)→ 실행` (컴파일=javac, 실행=JVM. 처음엔 둘 다 JVM이라 오해→교정) |
| 컴파일/실행에 필요한 것 | 컴파일 = **JDK**, 실행만 = **JVM/JRE** |
| `System.out.println("Hello, " + name)` → TS | `console.log(`Hello, ${name}`)` |
| `main`의 역할 | 프로그램의 **진입점(entry point)** |
| `.class`를 텍스트로 열면? | 못 읽음 — 바이너리(바이트코드). (`javap -c`로 역어셈블은 가능) |

**졸업 확인:** `kotlinc App.kt`로 만든 `App.class`를 친구 윈도우 PC에 보내 실행됨
→ 친구 PC엔 **JRE(=JVM 포함)** 필요, **JDK는 불필요**(컴파일은 내 PC에서 끝남). = Docker에 JRE만 넣는 이유.

## 핵심 한 줄 요약
- **컴파일은 javac/kotlinc(JDK)가, 실행은 JVM(⊂JRE)이 한다.**
- **컴파일 = 번역 전체 과정 / 파싱 = 그 안에서 토큰을 문법 트리로 조립하는 단계.**
- **`main` = 프로그램 진입점.** JDK는 스펙 → 배포판은 **Temurin** 사용.
