## ① WHY — "이거 왜 이렇게 길어?"

필드 2개짜리 데이터 클래스가 전통적 Java로는 ~20줄(필드 private + 생성자 + getter/setter 수작업).
TS면 `class User(public readonly name: string, public age: number)` 수준. 이 "장황함과 낯선 규칙들"의
정체를 알아야 남의 Java 코드를 막힘 없이 읽고, Kotlin이 왜 깔끔한지도 이해된다.

---

## ② WHAT — 개념 (TS 매핑 + Kotlin의 답)

### (a) 클래스 / 상속 / 인터페이스
| 개념 | TS | Java |
|---|---|---|
| 상속 | `extends` (제한 없음 비슷) | `extends` — **클래스 다중상속 불가**(하나만) |
| 인터페이스 구현 | 암묵적 | `implements A, B` — **여러 개 가능**, 명시 필수 |

**핵심 차이 — 명목적 vs 구조적 타이핑:**
- TS = **구조적**: 모양만 같으면 같은 타입 (자동 통과).
- Java/Kotlin = **명목적**: `implements`로 "이 계약 따른다"를 **명시 선언**해야 인정.
- Trade-off: 구조적은 유연하나 의도치 않은 매칭 위험 / 명목적은 장황하나 의도가 코드에 박혀 안전.

### (b) Java 관용구 3종 + Kotlin의 답
| Java 관용구 | 불편 | Kotlin | TS |
|---|---|---|---|
| getter/setter 보일러플레이트 | 필드마다 수작업 | `data class`/프로퍼티로 자동(+equals/hashCode/toString/copy) | 이미 간단 |
| **체크 예외**(`throws IOException`) | 호출자에게 try-catch/전파 **강제** | **폐지**(모두 unchecked) | 없음(unchecked) |
| 제네릭 소거(type erasure) | 런타임에 `List<String>`의 타입 인자 사라짐 | `reified`로 일부 우회 | TS도 소거 — 친숙 |

## ③ 미션 복기 (읽기 연습) + 교정

읽은 코드: `interface Greeter { String greet(String) throws IOException }` 와 그 구현 `EnglishGreeter`.

| 미션 | 결론 / 교정 |
|---|---|
| `implements` 꼭 쓰는 이유 | Java는 명목적 → 선언해야 계약 인정. TS면 구조적이라 자동. ✅ |
| `getPrefix()` 왜 있나 | Java는 getter 수작업. Kotlin `data class`/프로퍼티로 자동. ✅ |
| `throws IOException` 의미 | ⚠️ **인자 아님!** "이 메서드는 IOException 던질 수 있음" 표시. 호출자는 **try-catch로 잡거나 throws로 전파**를 강제당함(= 체크 예외). 안 하면 컴파일 에러. TS엔 이 강제 없음. |
| `@Override` | 부모/인터페이스 메서드를 재정의한다는 **컴파일러 검증용 표식**(오타 방지). TS 4.3+ `override` 키워드와 동일 역할. 데코레이터(코드 실행)와는 다름. |
| Kotlin이면 짧아질 부분 | getPrefix 제거 ✅, public 생략 ✅(기본 public). ⚠️ **반전: `override`는 사라지지 않고 필수 키워드가 됨.** + 생성자/필드/getter 한 줄로 합쳐짐 + `throws` 사라짐. |

### Kotlin 재작성 (20줄 Java → 6줄)
```kotlin
interface Greeter {
    fun greet(name: String): String          // throws 없음 (체크 예외 폐지)
}

class EnglishGreeter(private val prefix: String) : Greeter {  // 필드+생성자+getter 한 줄
    override fun greet(name: String): String =                // override 필수
        "$prefix, $name!"                                     // 문자열 템플릿(TS 백틱과 동일)
}
```
사라진 것: 생성자 본문, getter, `public`, `throws`. 강해진 것: `override` 필수.

## 핵심 한 줄 요약
- Java/Kotlin은 **명목적 타이핑**(implements 명시) — TS의 구조적과 다름.
- Java 장황함 3종(getter/setter, 체크 예외, 제네릭 소거) 중 **Kotlin은 getter자동·체크예외폐지**로 덜어냄.
- `throws X` = 체크 예외(호출자에 처리 강제), `@Override`/`override` = 재정의 표식(Kotlin은 필수).
- Kotlin 철학: **반복되는 일은 언어가 대신, 실수날 일은 더 강하게 강제.**
