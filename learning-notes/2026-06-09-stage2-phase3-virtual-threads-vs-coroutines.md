# Stage 2 · Phase 3-E ③ 학습 노트 — 가상 스레드 vs 코루틴 (JDK 21 Loom)

> 날짜: 2026-06-09
> 주제: Project Loom 가상 스레드, 코루틴/WebFlux와의 차이, "왜 신규 레포는 코루틴 대신 JDK 21을 쓰나"
> 페르소나: TypeScript 개발자 → 동기 코드 스타일로 비동기 처리량 얻기
> 계기: 실무 레포(`test-drive-portal-api`)를 JDK 20 → 21로 올리면서 생긴 의문 정리

---

## ① WHY — 같은 문제, 세 번째 해법

Phase 3-E ①②에서 본 그림:
- **블로킹 스레드**: 코드는 쉬운데(위→아래) OS 스레드가 무거움(~1MB) → 수만 동시 불가
- **코루틴**: `delay`가 스레드를 안 막아서 가벼움 → 처리량 ↑, **단 `suspend`가 호출 체인 전체로 전염**

그런데 2018년부터 JVM이 직접 푼 세 번째 길이 있다 → **가상 스레드(Project Loom, JDK 21 GA)**.
한 줄로: **"블로킹 코드를 한 글자도 안 바꾸고, 스레드만 가볍게 만든다."**

## ② WHAT — 핵심: "캐리어 스레드 양보"

```kotlin
// 코드는 평범한 블로킹. suspend 없음. async/await 없음.
fun handle(): Result {
    val a = db.query()    // 여기서 멈춤(블로킹)
    val b = api.call()    // 여기서도 멈춤
    return combine(a, b)
}
```
- 이걸 **가상 스레드 위에서** 돌리면: 블로킹되는 순간 JVM이 그 밑의 **진짜 OS 스레드(= 캐리어)를 놓아주고** 다른 가상 스레드를 올림.
- 가상 스레드 자체는 Heap에 사는 가벼운 객체(~수백 byte) → **수십만 개** 생성 가능.

> 🔗 Phase 3-E ① 심화와 같은 원리: 코루틴의 Continuation도 Heap에 산다. **가상 스레드는 그 "Continuation 저장→스레드 양보" 메커니즘을 JVM 레벨로 내려서, `suspend` 키워드 없이 모든 블로킹 호출에 자동 적용한 것.**

### 세 방식 한 장 비교

| | 블로킹 스레드 | 코루틴 / WebFlux | 가상 스레드 (JDK 21) |
|---|---|---|---|
| 코드 스타일 | 동기 (쉬움) | 비동기 (`suspend`/`Mono`) | **동기 (쉬움)** |
| 처리량 | 낮음 | 높음 | 높음 |
| "전염성" | 없음 | **있음** (체인 전체 suspend) | **없음** |
| JPA(블로킹 JDBC) | ✅ | ❌ (R2DBC 필요) | ✅ |
| 기존 코드 재사용 | — | 전면 재작성 | **거의 그대로** |
| 디버깅/스택트레이스 | 쉬움 | 깨짐 | 쉬움 |
| 켜는 법 | (기본) | 아키텍처 전환 | `spring.threads.virtual.enabled=true` |

### TS 개발자 매핑
- TS/Node는 단일 이벤트 루프라 "동기 블로킹 코드"라는 선택지가 애초에 없음 → 항상 async.
- JVM 가상 스레드는 **"Node의 처리량을 동기 코드 스타일로"** 얻는 느낌. async 색칠(coloring) 문제\*가 없다.
  - \*함수 색칠: async 함수는 async에서만 호출 가능 → 전염. 코루틴 `suspend`도 같은 문제. 가상 스레드엔 없음.

## ③ HOW — 왜 신규 레포는 코루틴 대신 JDK 21로 갔나 (실무 결정)

실제 `test-drive-portal-api` 스택을 보면 결정 이유가 명확:
- `spring-boot-starter-web`(Tomcat, **블로킹 MVC**) + `data-jpa`(**블로킹 JDBC**)
- WebFlux가 의존성엔 있지만 **`WebClient` 외부호출 용도뿐**, 그것도 `.block()`으로 동기 소비

→ 코루틴/WebFlux로 가려면 **JPA를 버리고 R2DBC로, 컨트롤러 전부 `suspend`로** 갈아엎어야 함.
→ 가상 스레드는 `languageVersion 20 → 21` + 설정 한 줄. **같은 처리량 목표를 1/100 비용으로.**

```kotlin
// build.gradle.kts
java { toolchain { languageVersion = JavaLanguageVersion.of(21) } }  // 20 → 21
```
```yaml
# application.yml — 가상 스레드까지 켤 때
spring.threads.virtual.enabled: true
```

## ⚠️ 트레이드오프 — "공짜 점심" 아님 (시니어 포인트)

가상 스레드를 *켜는 것*은 쉬워도, *효과 보는 것*은 별개다.

1. **DB 커넥션 풀이 진짜 병목**
   - 가상 스레드는 수십만 개 만들어도, Hikari pool이 10개면 결국 커넥션 10개에서 줄 섬.
   - DB 바운드 워크로드(이 레포처럼 거의 모든 요청이 JPA)는 **가상 스레드만으로 처리량이 안 늘 수 있다.** pool 튜닝 + DB 부하 재검토가 따라와야 함.
2. **`synchronized` pinning**
   - 가상 스레드가 `synchronized` 블록 안에서 블로킹하면 캐리어 스레드에 *고정*돼 양보 불가 → 이점 소멸.
   - JDK 21의 알려진 한계(JDK 24에서 해소). 구형 드라이버/라이브러리가 내부적으로 쓰면 영향. → `ReentrantLock`으로 교체 권장.
3. **ThreadLocal 남용**
   - 가상 스레드가 대량 생성되므로 무거운 ThreadLocal은 메모리 부담. (Spring Security context 등 표준 사용은 OK)

### 의사결정 표 (실무에 그대로 적용)

| 액션 | 권장 | 이유 |
|---|---|---|
| JDK 20 → 21 (toolchain만) | 👍 추천 | LTS 전환, 비용 ≈0, 동작 변화 없음 |
| 가상 스레드 켜기 | 🤔 조건부 | 켜기는 한 줄. 단 DB pool 병목이라 **부하 테스트로 before/after 검증 후** 결정 |
| 코루틴/WebFlux 전환 | ❌ 비추천 | 아키텍처 전면 재작성 >>> 이득. 가상 스레드가 더 싸게 같은 문제 해결 |

## 코루틴이 여전히 이기는 경우 (둘은 대체재 아님)
가상 스레드는 "스레드를 싸게" 만들 뿐, **동시성 구조를 주지 않는다.** 코루틴은 그 위에:
- **구조적 동시성** (`coroutineScope`, 자식 취소/에러 자동 전파)
- **취소(cancellation)**, `Flow`(스트림/백프레셔), `select`
→ 즉 "한 요청 = 한 스레드" 처리량 문제는 가상 스레드가, **복잡한 비동기 흐름 제어**는 코루틴이 강함.
실제로 코루틴을 가상 스레드 디스패처(`Dispatchers.LOOM` 류) 위에 얹는 조합도 가능.

## 핵심 한 줄 요약
- 가상 스레드 = "블로킹 코드 그대로 + 스레드만 가볍게". `suspend` 전염 없이 처리량을 얻는 JVM 레벨 해법.
- 신규 레포가 코루틴 대신 JDK 21을 택하는 이유 = **JPA/MVC 기존 자산을 안 버려도 되기 때문.**
- 단, **DB 커넥션 풀이 병목이면 켜도 효과 없음** → 부하 테스트로 검증하는 게 진짜 일.
